# repo-to-resume

代码仓库 → STAR 简历 → 模拟面试，求职者的项目加速器。

## 快速上手

**这个技能能做什么？** 分析你的项目代码，自动生成面试用的 STAR 简历、学习路径，还能模拟面试拷打。

**怎么用？** 直接告诉我就行，比如：

| 你想做什么 | 你可以这样说 |
|-----------|------------|
| 分析本地项目 | 「帮我分析 `C:\work\my-project` 这个项目」 |
| 分析 GitHub 仓库 | 「分析 https://github.com/xxx/yyy」 |
| 生成面试简历 | 「给这个项目写一份 STAR 简历」 |
| 准备项目面试 | 「我明天要面试，帮我准备这个项目的面试」 |
| 快速理解陌生代码 | 「我刚接手这个项目，帮我快速看懂」 |
| 只分析某一部分 | 「只看 `src/core/` 目录」「只看支付相关的代码」 |

**流程**：先圈定范围 → 生成简历 → 学习路径 → 模拟面试。每步都会跟你确认，你可以随时说「只看 XX」「跳过这步」「重新生成」。

## 触发条件

用户以任意方式表达分析项目意图时触发：
- 自然语言：「帮我分析这个项目」「给这个项目写简历」「准备项目面试」
- 贴路径或 Git URL
- @Skill 显式调用

## 输入

- 本地项目路径
- Git 仓库 URL（自动 clone 到用户指定目录）
- 压缩包路径

纯靠代码分析，不依赖 README 或外部文档。全语言通用，AI 自动识别技术栈。

## 对话输出格式规范（强制）

**设计目标**：阻断点输出不是"指令执行日志"，而是"设计过的交付卡片"。每个步骤的输出有清晰的视觉层次——文件路径是标题、关键指标是核心、详细内容可折叠。

### 一、卡片结构

每个阻断点输出遵循统一骨架：

```
> {emoji} {动作完成} · `{文件名}` · {1-2 个核心指标}

{关键指标——用列表或单行标签呈现，不用表格套列表}

{折叠的完整正文}

{引导性尾巴——一句话建议或提示}
```

### 二、折叠与标题降级

1. **折叠优先**：完整文档正文用 `<details><summary>📄 点击展开 XXX</summary>` 包裹，摘要直接展示在折叠外。
2. **标题降级**：折叠内的所有 Markdown 标题层级统一 +2（文件中的 `#` → 对话中 `###`，`##` → `####`，以此类推）。文件内保存的层级不变。

### 三、Emoji 语言（统一约定）

| 场景 | Emoji | 用途 |
|------|-------|------|
| 分析产出 | 📊 | Step 0 项目分析 |
| 简历产出 | 📄 | Step 1 STAR 简历 |
| 学习路径 | 📖 | Step 2 学习路径 |
| 交互页面 | 🖥️ | Step 2.5 交互式学习 |
| 面试报告 | 🎤 | Step 3 模拟面试 |
| 评分档位 | 🟢 🟡 🟠 🔴 | 85+ / 70-84 / 55-69 / <55 |
| 强调/必读 | 🔥 | 必读模块、高优先行动 |
| 警告/未通过 | ⚠️ | 自检未通过项 |
| 引导/建议 | 💡 | 一句话建议 |
| 反思/自评 | 💬 | 自我反思引导 |

### 四、表格使用原则

- 二维对照数据（维度→内容）→ 用表格
- 列表型数据（亮点、模块、行动项）→ 用 markdown 列表，**禁止**塞进表格导致首列为空
- 表格内不嵌套多行内容——一个单元格一句话

## 工作流（四步走，每步确认）

```
Step 0: 项目概览 → 深度分析（6 维度）→ ⛔ 确认 → STAR 简历
    ↓ 用户确认
Step 1: STAR 简历（一页纸 + 自检清单 + 重写机制）
    ↓ 用户确认
Step 2: 学习路径（三层递进：全景图 → 模块详解 → 跨模块面试题）
    ↓ 用户确认
Step 2.5: 交互式学习（单文件 HTML，浏览器打开按模块学习 + 随堂测验）
    ↓ 用户确认
Step 3: 模拟面试（多轮深挖 + 详细报告）
```

### Step 0：项目概览与深度分析

调用子模块 `references/code-analysis-engine.md` 执行全量分析，分两个阶段：

**阶段一：快速概览** — 文件列表获取、目录结构、语言占比统计、技术栈识别（从依赖文件推断）、配置文件分析。

**阶段二：深度分析（6 维度）** — 按「做什么 → 怎么做的 → 哪里容易出问题」的认知顺序展开：产品理解 → 角色识别 → 数据流 → 技术选型理由 → 巧妙模式 → 坑点。每个维度的具体分析方法和格式见 `references/code-analysis-engine.md`。

**分析深度**：文件级（读每个模块核心文件前 50-100 行）。大项目自适应：小型全量、中型模块级、大型关键路径级。

**信息来源**：允许读 README 和文档辅助理解，代码优先、文档为辅。

**失败处理**：深度分析失败时换策略重试一次，仍失败降级为轻量概览 + 标注「深度分析不完整」。

**输出**：先 `mkdir -p` 创建 `{项目路径}/{项目名}-analysis/` 子目录，再写入 `project-analysis.md`（固定模板，后续步骤直接引用）。

⛔ STOP — 阻断点（请务必执行以下步骤，缺一不可）

深度分析完成后，按以下卡片格式输出。**不要使用阿拉伯数字编号**——用小节标题替代：

```
> 📊 分析完成 · `project-analysis.md`

### 📊 项目概览

| 维度 | 内容 |
|------|------|
| 项目名 | {项目名} |
| 语言占比 | TypeScript 70% / Go 15% / ... |
| 文件数 | ~X 个源文件 |
| 技术栈 | React + Vite + Electron + Go CLI + ... |
| 架构模式 | 功能注册表 + Electron 多进程 + Go 子进程 |

### 📖 项目概述

> {项目名} 是一个 {一句话定位}，解决 {核心业务问题}。基于 {技术栈} 构建，采用 {架构模式}，核心包含 {2-3 个关键模块}。

### 🔍 六维度摘要

| 维度 | 要点 |
|------|------|
| 产品理解 | {一句话定位 + 核心功能} |
| 角色识别 | {5-8 个核心模块及职责} |
| 数据流 | {1-2 条端到端路径} |
| 技术选型理由 | {3-5 个关键选型及理由} |
| 巧妙模式 | {3-5 个工程亮点} |
| 坑点 | {2-3 个已知问题} |

💡 建议覆盖 `src/core/`，约 15 个文件
```

以上内容输出完毕后，**停下来，等待用户回复**——在用户明确确认之前，**禁止**执行 Step 1 的任何操作（包括读取 star-resume-generator.md、生成简历、写入 resume.md）。此项为内部指令，**不要输出给用户**。

用户确认后，主动衔接：「分析完成。是否开始生成 STAR 简历？你也可以说「只看 XX 部分」缩小范围。」

### Step 1：生成 STAR 简历

调用子模块 `references/star-resume-generator.md`，量化估算规则参考 `references/star-framework.md`。

**输出格式**：Markdown 或 HTML，用户可选。

核心要求（详见子模块）：
- 成果导向：每条 Action 回答「业务问题 + 技术决策 + 量化结果」
- 强动词开头：设计/主导/重构/消除/降低/建立；避免负责/参与/协助/维护
- 技术关键词可检索：技术栈自然嵌入正文（面试官 Ctrl+F 能搜到）
- 删除主观评价：不出现「深刻理解」「熟练掌握」「深度」「全面」「显著」等空洞形容词
- 一页以内：不超过 10 条 Action，每条 ≤80 字；取舍标准——这条能不能让面试官 5 秒内决定「这个人值得聊」？
- 估算值统一声明：量化结果不分类，简历末尾统一标注「建议替换为真实运行数据」

生成后按子模块自检清单逐条检查，不通过自动重写（最多 2 次），仍不达标标注 `【需人工优化】`。

**边界**：
- 自动跳过：密钥/token/密码、node_modules/vendor 等依赖目录、二进制/图片/视频等非代码资源
- 忽略测试代码
- 不分析 Git 历史
- 简历开头显眼位置标注「AI 生成，建议人工复核」

⛔ STOP — 阻断点（请务必执行以下步骤，缺一不可）

简历产出后，按以下卡片格式输出。**不要使用阿拉伯数字编号**：

```

<details><summary>📄 点击展开完整简历</summary>

（在此嵌入简历正文，标题已降级——# → ###，## → ####）

</details>

📝 **{X} 条 Action** · 架构决策层 {X} / 工程深度层 {X} / 质量工程层 {X}

**关键亮点**
1. {亮点1（≤30 字）}
2. {亮点2（≤30 字）}
3. {亮点3（≤30 字）}
（最多 5 条）

⚠️ 自检未通过：{xxx}（如全部通过则省略此行）
```

以上内容输出完毕后，**停下来，等待用户回复**——在用户明确说「满意」「继续」「下一步」之前，**禁止**执行 Step 2 或 Step 3 的任何操作。此项为内部指令，**不要输出给用户**。

用户确认后，主动衔接：「简历已生成。接下来可以：① 生成学习路径，按模块拆解代码帮你深入理解；② 直接开始模拟面试，基于简历亮点追问。你想继续哪一步？」

### Step 2：生成学习路径

调用子模块 `references/learning-path-generator.md`。

采用三层递进结构，解决旧版"模块孤立、无重点"的问题：

1. **系统全景图**：ASCII 依赖图 + 角色表（标注重要度 ⭐） + 一条请求走到底。用户先建立心智模型，知道模块之间怎么串。
2. **模块详解**：核心模块（⭐⭐⭐）完整展开（代码↔白话对照 + 设计决策分析），支撑模块（⭐）精简呈现。用户知道把 70% 时间花在哪。
3. **跨模块面试题**：数据流追踪 + 架构权衡 + 故障隔离——这些才是面试官最可能问的系统视角题。

详细模板和生成规则见 `references/learning-path-generator.md`。

⛔ STOP — 阻断点（请务必执行以下步骤，缺一不可）

学习路径产出后，按以下卡片格式输出。**不要使用阿拉伯数字编号**：

（在此嵌入学习路径正文，标题已降级——# → ###，## → ####）

</details>

模块分布：⭐⭐⭐ 核心 ×{X} · ⭐⭐ 重要 ×{X} · ⭐ 支撑 ×{X}

**🔥 必读模块**
1. **{模块名}** — {为什么必读（≤20 字）}
2. **{模块名}** — {为什么必读（≤20 字）}
（最多 3 个）

**核心亮点**：{一条请求走到底的路线概述（≤50 字）}

**🎤 面试官会追问**
- 数据流：{题（≤30 字）}
- 架构权衡：{题（≤30 字）}

```

以上内容输出完毕后，**停下来，等待用户回复**——在用户明确说「继续」「下一步」之前，**禁止**执行 Step 2.5 或 Step 3 的任何操作。此项为内部指令，**不要输出给用户**。

用户确认后，主动衔接：「学习路径已生成。接下来可以：① 生成交互式学习页面，在浏览器中按模块学习（推荐）；② 直接开始模拟面试。你想继续哪一步？」

### Step 2.5：交互式学习

**触发条件**：Step 2 用户确认后，如果用户说「继续」「下一步」「生成学习页面」「开始学习」等，执行此步。如果用户明确说「跳过学习」「直接面试」，跳到 Step 3。

**目的**：把 Step 2 产出的 markdown 学习路径转换为交互式 HTML，用户在浏览器中按模块学习，核心模块附带随堂测验。解决"读完 markdown 还是记不住"的问题。

调用子模块 `references/interactive-learning-template.md`。

**核心约束**：
- CSS 变量和 JS 脚本**必须原样复制**，不得修改任何颜色、字号、间距——确保每次生成的 HTML 视觉一致
- 内容从 Step 2 产出的 `learning-path.md` 中提取，不重新生成内容
- 测验题从模块的「面试可能追问」中衍生，考应用而非记忆（「如果要加 X 功能，改哪个模块？」而非「X 模块做了什么？」）

**输出**：单文件自包含 HTML，写入 `{项目路径}/{项目名}-analysis/learning-interactive.html`。外部依赖仅 Google Fonts CDN，双击即开。

⛔ STOP — 阻断点（请务必执行以下步骤，缺一不可）

HTML 生成后，按以下卡片格式输出。**不要使用阿拉伯数字编号**：

```
> 🖥️ 交互式学习页面已生成 · `learning-interactive.html`

在文件管理器中双击即可在浏览器中打开。

| 模块 | {X} 个（含全景图） |
| 随堂测验 | {核心模块名列表} |

💡 按顺序浏览（键盘 ↑↓ 切换模块），完成所有测验后再进入模拟面试效果最好。
```

以上内容输出完毕后，**停下来，等待用户回复**——在用户明确说「学完了」「继续」「开始面试」之前，**禁止**执行 Step 3 的任何操作。此项为内部指令，**不要输出给用户**。

用户回复后，主动衔接：「学完了吗？准备好模拟面试了吗？我会基于简历亮点和跨模块关系逐题追问，帮你查漏补缺。」

### Step 3：模拟面试

调用子模块 `references/mock-interviewer.md`，通用题库见 `references/interview-question-templates.md`。

采用四阶段结构化面试，解决旧版「一口气问到底、难度不分层、评分模糊」的问题：

1. **暖场**（~10%）：项目介绍 + 最满意的一段代码——让用户放松，建立对话节奏
2. **技术深挖**（~50%）：代码出题 + 简历亮点深挖，按回答质量决定追问深度（澄清→深挖→质疑→扩展），不靠计数。连续 3 题薄弱时暂停询问是否降难度
3. **系统视角**（~25%）：跨模块数据流追踪 + 架构权衡 + 场景假设——区分「能写代码」和「能设计系统」
4. **收尾**（~15%）：反思题 + 「你有什么想问我的？」——展示主动性

**评分**：五维度行为锚定（每个分数段有可观测的行为描述），不再是模糊的「回答准确深入」。

**衔接**：如果 Step 2.5 已执行，阶段 2 开场时从测验薄弱点切入深挖。

核心规则详见 `references/mock-interviewer.md`（四阶段流程 / 追问梯度 / 行为锚定评分表 / 行动清单模板 / 项目画像→题库权重）。

⛔ STOP — 阻断点（请务必执行以下步骤，缺一不可）

面试结束后，按以下卡片格式输出。**不要使用阿拉伯数字编号**。档位标签从评分行为锚定表取：

```
> 🎤 面试结束 · `interview-report.md` · {N} 题 · 约 {M} 分钟

<details><summary>📄 点击展开完整面试报告</summary>

（在此嵌入面试报告正文，标题已降级——# → ###，## → ####）

</details>

| 总分 | **{X}/100** · {🟢🟡🟠🔴 档位标签} |
|------|----------------------------------|
| 最强 | {维度名} {X}/20 |
| 最弱 | {维度名} {X}/20 |

暖场 {X} 题 · 深挖 {X} 题 · 系统 {X} 题 · 收尾 {X} 题

**🔴 优先复习**
1. {任务描述（≤30 字）}
2. {任务描述（≤30 字）}
（最多 3 条）

💬 报告中包含 3 个自我反思问题，建议花 2 分钟回答
```

以上内容输出完毕后，**停下来，等待用户回复**——在用户明确回复之前，**禁止**自动开始新一轮面试或重新生成。此项为内部指令，**不要输出给用户**。

用户回复后，主动衔接：「面试结束。报告里有一份按优先级排列的复习行动清单，每条都有具体步骤和达标标准——建议从高优先的开始逐条完成。完成后可以再来一轮面试检验效果。要再来一轮吗？或者你想重新生成某一步的产出？」

## 输出目录结构（强制）

**所有产出必须保存到项目目录内的 `{项目名}-analysis/` 子目录中。**

目录创建规则：
- 以被分析项目的**根目录**为基准，在其内部创建 `{项目名}-analysis/` 子目录
- 例如分析 `C:\work\my-project`，则产出目录为 `C:\work\my-project\my-project-analysis/`
- **在写入任何文件之前，必须先 `mkdir -p` 创建该目录**
- 每个 Step 写入文件前都必须确认目录已存在
- **同名文件版本管理**：写入前检查目标文件是否已存在。若已存在，新文件自动追加版本后缀（如 `resume-v1.md`、`resume-v2.md`），取当前目录下已有同名文件的最大版本号 +1。禁止覆盖已有文件。

```
{项目名}/
├── src/                       # 被分析的项目源码
├── package.json
├── ...
└── {项目名}-analysis/         # 所有产出（项目内的子目录）
    ├── project-analysis.md    # 项目深度分析报告（Step 0 产出）

## 架构

主 Skill 负责编排流程，5 个子模块各司其职：

| 子模块 | 职责 | 文件 |
|--------|------|------|
| 代码分析引擎 | 扫描结构、识别架构、深度分析（6 维度）、提取亮点 | `references/code-analysis-engine.md` |
| STAR 简历生成器 | 套 STAR 框架填内容 | `references/star-resume-generator.md` |
| 学习路径生成器 | 三层递进：全景图 → 模块详解 → 跨模块面试题 | `references/learning-path-generator.md` |
| 交互式学习模板 | 单文件 HTML 模板 + CSS/JS（固定不可改）+ 测验逻辑 | `references/interactive-learning-template.md` |
| 模拟面试官 | 出题 + 追问 + 评分 + 报告 | `references/mock-interviewer.md` |

辅助参考：
| 量化估算规则 | 如何从代码特征推算合理量化指标 | `references/star-framework.md` |
| 通用题库 | 按技术栈匹配的面试题模板 | `references/interview-question-templates.md` |

数据传递：结构化数据（分析结果）由主 Skill 传递，长文本（简历、学习路径、报告）用文件传递。`project-analysis.md` 作为 Step 0 产出，供 Step 1 简历生成和 Step 3 模拟面试直接引用。

## 失败模式与恢复

以下场景必须显式处理，不得静默失败：

| 触发条件 | 一线修复 | 仍失败则 |
|---------|---------|---------|
| Git clone 失败（URL 无效/无网络/无权限） | 提示用户检查 URL 和网络，建议手动下载后提供本地路径 | 终止，告知用户具体错误原因 |
| 本地路径不存在或不可读 | 提示用户检查路径拼写和权限 | 终止 |
| 压缩包解压失败（损坏/格式不支持） | 提示用户检查文件完整性，建议重新下载 | 终止 |
| 项目目录为空（无代码文件） | 提示用户「目录中未发现代码文件，请确认路径」 | 终止 |
| 无法识别语言/技术栈 | 告知用户「未能自动识别技术栈」，尝试通用文件结构分析 | 降级为仅输出目录结构和文件清单 |
| 分析超时（项目过大或文件过多） | 自动降级：仅分析顶层 2 层目录，跳过文件内容扫描 | 输出已分析部分 + 标注「分析不完整」 |
| 用户指定的范围无匹配文件 | 提示「指定范围内未找到匹配文件」，展示实际目录结构供重新选择 | 回到 Step 0 重新圈定范围 |
| 模拟面试中用户连续 3 题答不上来 | 暂停深挖，询问「要降低难度还是继续？」 | 切换为讲解模式，逐题给参考答案 |

## 边界约束

- 纯代码分析，不依赖外部信息
- 不碰敏感信息（密钥、token、密码）
- 不分析第三方依赖源码（node_modules、vendor 等）
- 不分析二进制/图片/视频等非代码资源
- 忽略测试代码
- 不分析 Git 历史
- 独立运行，不依赖其他 Skill
- 大项目（>10 万行）先提示用户缩小范围
- 简历开头标注 AI 生成声明

## 反例：不要做的事

- ❌ 不要在用户未确认范围时直接开始全量分析
- ❌ 不要分析 node_modules、vendor、.git 等目录
- ❌ 不要在简历中输出密钥、token、密码等敏感信息
- ❌ 不要编造量化数据——必须标注「估算值」和真实数据来源
- ❌ 不要在模拟面试中一次抛出多个问题
- ❌ 不要跳过用户确认直接进入下一步
- ❌ 不要在面试报告中评价用户个人，只评价回答质量
- ❌ 不要分析测试代码和 Git 历史

## 用户确认与修改

每步产出后，用户可确认或提出修改意见：
- 「满意，继续下一步」
- 「{具体修改要求}」→ AI 修改后重新确认
- 产出文件可直接手动编辑，也可通过对话让 AI 改
