# STAR 简历生成器

基于代码分析结果，生成一页纸 STAR 简历。

## 输入

- 代码分析引擎的结构化输出（JSON）
- 用户指定的输出格式（Markdown / HTML）

## 输出格式

### Markdown 模板（保持此结构——面试官习惯这个信息层次，偏离会降低可读性）

```markdown
# {项目名称}

> 🤖 本文由 AI 生成，建议人工复核后使用。

## Situation（项目背景）

{一句话说清项目解决了什么业务问题}

{项目定位与核心功能，1-2 句话}

**技术栈**：{语言} + {框架} + {数据库} + {中间件}（按层次排列，每个技术点必须在 Action 中出现过）

## Task（工作职责）

{一句话描述你在项目中的具体职责，不是复读项目描述}

## Action & Result（核心工作与成果）

{不超过 10 条，每条 ≤80 字，允许两个句子；先写业务问题+技术决策，再写量化结果}

1. {做了什么}，{带来的效应}。
2. {做了什么}。
...
（不超过 10 条）

📊 以上为基于代码规模估算值，建议替换为真实运行数据。
```

## 六条核心原则（生成时必须遵守）

### 1. 成果导向，不是职责清单

面试官想看的是你造成了什么改变，不是你坐在什么位置上。每条 Action 回答三个问题：
- 解决了什么业务问题？
- 做了什么技术决策？
- 产生了什么可度量的结果？

模板：**通过[方法]，使[指标]从[X]提升/降低到[Y]**。

| ❌ 差 | ✅ 好 |
|-------|------|
| 负责后端 API 开发 | 重构订单系统 API，P99 延迟从 800ms 降到 120ms |
| 参与微服务拆分 | 主导用户模块从单体拆分为独立服务，支撑日活从 5 万到 30 万无扩容 |

### 2. 强动词开头

每行用强动词起手。禁止弱动词。

| ✅ 强动词 | ❌ 弱动词（禁止） |
|-----------|-------------------|
| 设计、主导、重构、消除、降低、建立、引入、实现 | 负责、参与、协助、维护 |

### 3. 技术关键词可检索

面试官和 ATS 都是 Ctrl+F 搜的。技术栈自然嵌入 Action 正文，不单独列「技能」表格。

| ❌ 差 | ✅ 好 |
|-------|------|
| 技能：Redis, Kafka | 基于 Redis + Kafka 实现异步任务队列，消息积压从 10 万降至 500 |
| 技术栈：Pydantic v2 | 引入 Pydantic v2 构建类型安全配置层，消除 50+ 配置项的运行时类型错误 |

### 4. 删除所有主观评价

用事实替代形容词。以下词汇**禁止出现**：

`深刻理解` `熟练掌握` `良好的编码习惯` `深度` `全面` `显著` `极佳` `大量` `充分` `强大` `优雅` `完美` `极致`

| ❌ 差 | ✅ 好 |
|-------|------|
| 深入理解 JVM 内存模型 | 通过 GC 调优减少 Full GC 次数 90% |
| 具备良好的编码习惯 | （删除，用具体实践替代） |

### 5. 一页以内

- 不超过 10 条 Action，每条 ≤80 字
- 取舍标准：**这条能不能让面试官 5 秒内决定「这个人值得聊」？** 不能就删

### 6. 估算值统一声明

量化结果不分类不标置信度，简历末尾统一写一段提示：

> 📊 以上为基于代码规模估算值，建议替换为真实运行数据。

如何从代码特征推算合理量化指标（并发模型→QPS、缓存策略→命中率等），见 `references/star-framework.md` 量化估算规则表。

## 生成规则

### Situation 生成

**三段式结构**（面试官期待先看到业务问题再看到技术方案，保持此顺序）：

1. **业务问题**（第一段）：一句话说清项目解决了什么业务痛点。AI 从项目结构反推草稿，Step 0 圈定范围时主动问用户确认。
2. **项目作用**（第二段）：项目定位与核心功能，1-2 句话。
3. **技术栈**（第三段，独立成段）：按层次排列（语言 → 框架 → 数据库 → 中间件 → 其他），每个技术点必须在 Action 中出现过。

示例：
> 项目需求模糊、测试覆盖不全导致版本交付质量差，每次验收出现大量逻辑 bug，上线后高频线上故障。
>
> WPS 自动化测试用例批量生成框架，将需求文档自动转化为可执行的桌面自动化测试用例，提供 Pipeline 编排、批量调度、知识检索、用例记忆等核心能力。
>
> **技术栈**：Python 3.11+ + Pydantic v2 + Jinja2 + aiohttp + SQLite FTS5 + Rust（knowledge-cli / case-validator）

### Task 生成

- 从代码模块推断用户角色
- 一句话描述**具体职责**，不是复读项目描述
- 示例：「负责业务应用层设计与开发，涵盖 Pipeline 编排与调度、资源约束的跨用例并行执行、多源知识检索注入、用例级记忆系统。」

### Action & Result 生成（核心规则）

#### 格式规则（遵循这些规则让每条 Action 信息密度最大化）

每条 ≤80 字，允许两个句子（用句号分隔）。先写业务问题+技术决策，再写量化结果：

```
{序号}. {业务问题}，{技术决策}，{量化结果}。
```

量化结果能写就写，没有不硬凑。预防性决策允许不写结果，但必须写清「如果不这样做会有什么风险」。

#### 分类与排序

| 类型 | 位置 | 要求 |
|------|------|------|
| **亮点类**（有量化结果） | 放前面 | 必须包含量化结果 |
| **工程素养类**（预防性决策） | 放后面 | 允许不写结果，但必须写清风险 |

#### 分层质检（内部约束）

AI 生成时必须确保三层都有覆盖，缺层时提示补充：

| 层级 | 说明 | 示例 |
|------|------|------|
| **架构决策层** | 证明系统设计能力 | 流水线调度器、熔断器、多载体适配 |
| **工程深度层** | 证明解决硬问题的能力 | FTS5 安全化、记忆系统、探针契约 |
| **质量工程层** | 证明关注可靠性 | 配置类型安全、并发显示、自检机制 |

#### 避免事项（这些会让简历看起来像项目文档而非面试材料）

- ❌ **禁止出现具体代码文件名**（如 `fill_prompts.py`、`validate_cases.py`、`SKILL.md` 等）
- ❌ **禁止出现代码块、ASCII 图、架构图、流程图**
- ❌ **禁止使用子标题、子列表、多级缩进**
- ❌ **禁止使用加粗分段**（如 `**架构设计**：xxx`）
- ❌ **禁止空洞形容词**（见禁用词表）
- ❌ **禁止出现目录路径**（如 `.cursor/skills/`、`assets/prompts/`）
- ❌ **禁止出现版本号**（如 `v5.0.0`）——除非是外部依赖的版本
- ❌ **禁止自造术语**——用业内通用表述替代（如「后台异步合并（基于 LLM 的摘要+标签提取）」而非「LLM 驱动的后台异步 Consolidator」）

## 生成后自检清单

简历生成后，AI 必须逐条自检。不通过的项目自动重写（最多 2 次），仍不达标标注 `【需人工优化】`：

- [ ] 每条 Action 是否回答了「业务问题 + 技术决策 + 量化结果」？
- [ ] 是否用强动词开头（设计/主导/重构/消除/降低/建立）？
- [ ] 是否避免了弱动词（负责/参与/协助/维护）？
- [ ] 技术栈关键词是否自然嵌入正文（Ctrl+F 可检索）？
- [ ] 是否删除了所有主观形容词（深刻理解/熟练掌握/深度/全面/显著等）？
- [ ] 是否不超过 10 条 Action，每条 ≤80 字？
- [ ] 三层（架构决策/工程深度/质量工程）是否都有覆盖？
- [ ] 有量化结果的亮点 Action 是否放在前面？
- [ ] 是否避免了代码文件名、代码块、目录路径、子标题？
- [ ] 是否避免了自造术语（用业内通用表述替代）？
- [ ] Situation 是否拆为三段（业务问题 → 项目作用 → 技术栈）？
- [ ] Task 是否写的是具体职责而非复读项目描述？

## 质量要求

- 整份简历控制在 1 页（约 500-800 字）
- 每条 Action ≤80 字，不超过 10 条
- 信息密度高：每条包含「业务问题 + 技术决策 + 量化结果」三要素
- 技术术语准确，与代码实际一致
- **简历读者是面试官，不是同事**——不要出现只有项目内部人才懂的命名
