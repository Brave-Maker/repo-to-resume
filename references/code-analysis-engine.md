# 代码分析引擎

分析项目代码仓库，输出结构化分析结果供后续子模块使用。

## 输入

- 项目根目录路径
- 用户指定的分析范围（目录/文件类型/关键词/自然语言描述）

## 分析策略

### 信息来源

允许读取 README、项目文档、代码注释辅助理解项目。代码优先，文档为辅——先读代码形成判断，再用文档交叉验证，不一致时以代码为准。

### 分析深度

文件级深度——读取每个模块的核心文件（前 50-100 行），理解关键类和函数。

### 大项目自适应

- 小型（<3 万行）：文件级深度，全量分析
- 中型（3-10 万行）：模块级深度，读每个模块的入口文件
- 大型（>10 万行）：关键路径级，只追踪 2-3 条核心业务流程的调用链

### 失败处理

深度分析失败时：
1. 自动换分析策略重试一次（如文件级降为模块级）
2. 仍失败则降级为轻量概览模式（仅目录结构 + 文件统计 + 技术栈识别），标注「深度分析不完整」

## 分析维度

### 阶段一：快速概览（轻量，用于后续深度分析的基础）

#### 1. 项目概览

- 目录树结构（深度限制 3 层）
- 文件统计：总文件数、各语言文件数及占比
- 代码行数估算（排除空行和注释）
- 项目规模判断：小型（<1 万行）/ 中型（1-10 万行）/ 大型（>10 万行）

#### 2. 技术栈识别

- 编程语言及版本（从配置文件推断）
- 框架及版本（如 Spring Boot、React、Django 等）
- 数据库/存储（从依赖或配置文件识别）
- 构建工具（Gradle、Maven、npm、pip 等）
- 关键依赖库（从依赖文件提取核心依赖）

#### 3. 架构分析

- 架构模式识别：MVC / 微服务 / 事件驱动 / 分层架构 / 插件式 等
- 模块边界：识别顶层模块/包，分析模块职责
- 模块间依赖关系：分析 import/include 关系
- 调用链路：识别核心业务流程的调用路径

#### 4. 入口点识别

- Web 应用：路由定义、Controller、中间件
- CLI 工具：main 函数、命令定义
- 库/SDK：公开 API、导出接口
- 服务：RPC 定义、消息队列消费者

### 阶段二：深度分析（6 维度，按认知顺序排列）

以下 6 个维度按「做什么 → 怎么做的 → 哪里容易出问题」的认知顺序排列，每个维度控制在 3-5 行。

#### 5. 产品理解

- 一句话产品定位（项目解决了什么业务问题）
- 核心功能列表（2-3 条）
- 目标用户或使用场景

#### 6. 角色识别

- 列出核心模块/组件名 + 一句话职责
- 格式：`模块名`：职责描述
- 覆盖 5-8 个核心模块，按重要度排序

#### 7. 数据流

- 描述 1-2 条核心业务流程的端到端数据流转路径
- 格式：`触发条件 → 模块A（处理X）→ 模块B（处理Y）→ 模块C（输出Z）`
- 每条路径控制在 3-5 步

#### 8. 技术选型理由

- 列出 3-5 个最关键的技术选型决策
- 每个附一句话选型理由 + 一句话替代方案（如果不用这个会用什么）
- 格式：`技术X`：选型理由 → 替代方案 `技术Y`

#### 9. 巧妙模式

- 列出 3-5 个工程亮点，每个一句话描述
- 聚焦：并发处理、缓存策略、设计模式、插件机制、错误处理、安全措施
- 格式：`模式名`：一句话描述其巧妙之处

#### 10. 坑点

- 列出 2-3 个已知问题或技术债，每个一句话描述
- 来源：代码中的 TODO/FIXME/HACK 注释、明显的防御性代码、硬编码配置
- 格式：`问题`：一句话描述影响

分析完成后，将结果返回给主 Skill（SKILL.md），由主 Skill 的 Step 0 STOP 阻断点控制确认流程和后续步骤。本模块只负责分析，不控制流程。

## 跳过规则

自动跳过以下内容：
- `node_modules/`、`vendor/`、`target/`、`build/`、`dist/`、`.git/` 等依赖和构建目录
- `*.min.js`、`*.bundle.js` 等压缩文件
- 二进制文件（`*.class`、`*.jar`、`*.exe`、`*.so`、`*.dll` 等）
- 图片/视频/音频文件
- 包含 `password`、`secret`、`token`、`key` 等敏感关键词的配置文件（跳过内容，仅记录文件名）
- 测试目录（`test/`、`tests/`、`__tests__/`、`spec/`）
- `.gitignore` 中标记为生成/忽略的文件
