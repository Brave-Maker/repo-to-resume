# STAR 法则框架说明

STAR 法则是一种结构化的经历描述方法，用于在简历和面试中清晰展示项目经验。

## 四要素

| 要素 | 含义 | 简历中的体现 |
|------|------|-------------|
| **S**ituation | 项目背景和上下文 | 项目是什么、解决什么问题、技术栈 |
| **T**ask | 你的角色和任务 | 你在项目中负责什么 |
| **A**ction | 你采取的行动 | 具体做了什么、怎么做的、技术决策 |
| **R**esult | 取得的成果 | 量化指标、业务价值 |

## Action 写作规范

每条 Action 必须包含三个要素：

```
{技术决策} + {实现方式} + {量化结果}
```

### 技术决策

说明你做了什么选择，体现判断力。

✅ 好的决策描述：
- 「引入 Java 21 虚拟线程替代平台线程」
- 「设计 Redis Stream 异步消费架构」
- 「封装 StructuredOutputInvoker 统一处理结构化输出」

❌ 差的决策描述：
- 「负责后端开发」
- 「写了很多代码」
- 「优化了性能」

### 实现方式

说明具体怎么做的，体现执行力。

✅ 好的实现描述：
- 「将 LLM 推理任务从请求链路解耦，通过 Redis Stream 消费组异步处理」
- 「失败时将错误原因注入 prompt 引导模型自修复」
- 「采用文档分块 + 1024 维向量化 + HNSW 索引」

❌ 差的实现描述：
- 「用了 Redis」
- 「做了优化」
- 「写了一些工具类」

### 量化结果

说明做到了什么程度，体现影响力。

✅ 好的量化结果：
- 「核心接口响应时间从 15s 降至 200ms」
- 「支撑千级并发」
- 「有效拦截无关回答」

❌ 差的量化结果：
- 「大幅提升性能」
- 「显著改善体验」
- 「效果很好」

## 量化估算规则

当无法获取真实数据时，基于代码规模进行合理估算：

| 代码特征 | 可估算指标 | 估算方法 |
|---------|-----------|---------|
| 并发模型（虚拟线程/协程/线程池） | 并发量 | 按线程数 × 10 估算 QPS |
| 缓存策略（Redis/Caffeine） | 命中率提升 | 按缓存层数估算，单层约 30-50% |
| 异步架构（消息队列/Stream） | 响应时间优化 | 按同步→异步估算，通常 10-100x |
| 批量处理 | 吞吐量提升 | 按批量大小估算，通常 5-20x |
| 索引优化 | 查询性能 | 按索引类型估算，通常 10-1000x |
| 限流/熔断 | 可用性 | 按降级策略估算，通常 99%→99.9% |

**每条估算必须附真实数据来源提示**：
- 「📊 估算值，真实数据建议从监控平台（Grafana/Prometheus）查看实际 QPS」
- 「📊 估算值，真实数据建议从 APM（SkyWalking/Pinpoint）查看实际响应时间」
- 「📊 估算值，真实数据建议从压测报告获取实际并发上限」
